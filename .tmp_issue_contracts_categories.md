## Summary
Define and generate contract support for event categories so backend and frontend can implement category persistence against a stable API.

## Problem
Categories exist in the database design and in frontend UX, but the current OpenAPI contract does not expose event category CRUD/read endpoints. There is also drift between contract and backend entry payloads around category fields.

## Scope
- Add event category schemas to the OpenAPI contract
- Add event category read shape to event details responses
- Add event category create/update/delete endpoints for event settings flows
- Reconcile category-related fields in entry request/response schemas with actual backend intent
- Regenerate contract artifacts and ensure `contracts:check` remains green

## Acceptance Criteria
- OpenAPI includes explicit category DTOs and event category CRUD endpoints
- Event details contract includes categories needed by FE settings
- Entry schemas consistently represent category references/names according to agreed design
- Generated TypeScript client includes the new category operations/models
- Contract validation passes with no drift

## Notes
- This issue is the dependency gate for backend and frontend implementation work on categories
- Backend implementation should depend on this issue being completed first
