## Summary
Add backend support for event categories so the existing frontend categories settings can persist to and read from the backend.

## Depends On
- #55

## Problem
The repository already has an `event_categories` database table and the frontend already has a categories settings UI, but there is no backend implementation for category CRUD/read and no category data is exposed in event details.

## Scope
- Add backend category model for event-scoped categories
- Expose category data in event details payloads
- Add category CRUD endpoints for event settings use cases
- Validate uniqueness and ownership/membership constraints
- Align implemented API with contracts from #55
- Ensure entry flows can reference persisted categories consistently

## Acceptance Criteria
- `GET /events/{eventId}` returns categories needed by FE settings and entry flows
- Backend exposes category create/update/delete operations for an event
- Default vs custom category state is persisted and returned explicitly
- Duplicate category names within the same event are rejected with clear validation errors
- Authorization rules match existing event membership rules
- Backend implementation matches generated contracts from #55
- Backend tests cover happy path, validation failures, duplicates, access control, and regression scenarios

## Notes
- Current FE categories UI already exists in event settings and should become the consumer of this API
- Current backend has only table-level support (`event_categories`) but no entity/repository/service/controller path
- Current contract/backend also show category-related drift in entry payloads that should be reconciled during this work
