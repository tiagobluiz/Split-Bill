## Summary
Link the existing event settings categories UI to backend category APIs once backend support is available.

## Depends On
- #53

## Problem
The frontend already renders editable categories in event settings, but the data is local-only and not persisted. After backend support exists, FE must stop using local category state and use the backend as the source of truth.

## Scope
- Replace local categories state in event settings with backend-backed queries/mutations
- Hydrate default and custom categories from the event details response
- Persist add/remove/edit interactions through backend APIs
- Keep alignment with Prompt 06 render and repository UX guidelines
- Ensure entry-related category selectors consume the same backend-backed category data where applicable

## Acceptance Criteria
- Event settings categories render from backend data only
- Adding/removing/editing categories persists immediately without a save button
- Loading, error, and empty states follow frontend UX guidelines
- Existing settings layout continues to match Prompt 06 expectations
- FE tests cover load, create, edit, delete, validation error, and backend error states
- No mock/local-only category persistence remains in the delivered flow

## Notes
- The current categories bubble already exists in FE and should be reused rather than redesigned
- This issue should not start until backend/category contract support from #53 is available
